# Các Thành Phần Cốt Lõi Của AI Agent

## 1. The Brain - Bộ Não (Foundation Model/LLM)

### Vai Trò
- Đóng vai trò như **CPU** (Central Processing Unit) của agent
- Là engine suy luận và nhận dạng mẫu (pattern recognition)
- Không lưu trữ bộ nhớ hay thực hiện hành động trực tiếp
- Điều phối tất cả các thành phần khác

### Cách Hoạt Động
- Được train trên massive, diverse data từ internet, sách, code
- Học các pattern thống kê của ngôn ngữ và logic
- Kỹ năng cơ bản: **Next Token Prediction** (dự đoán từ tiếp theo)
- Khi dự đoán được mở rộng, nó trông giống như **reasoning** (suy luận)

### Prompting Patterns Để Kích Hoạt Reasoning

#### Chain of Thought (CoT)
- Instruct LLM: "Think step by step" hoặc "Show your work"
- Agent tạo ra internal reasoning process trước khi đưa ra câu trả lời cuối
- Ví dụ: "Thời tiết ở London thế nào và tôi có nên mang ô không?"
  - Step 1: Cần thông tin thời tiết để đưa ra khuyến nghị
  - Step 2: Gọi API thời tiết, kết quả: light rain
  - Step 3: Light rain → nên mang ô
  - Step 4: Trả lời: "Yes, you should take an umbrella"

#### ReAct Pattern
- Kết hợp Reasoning + Acting trong một vòng lặp
- Liên tục: Thought → Action → Observation → Repeat

### Trade-off Khi Chọn LLM
Ba yếu tố cần cân nhắc:
1. **Chất lượng suy luận (Quality)**: Agent có thể theo dõi plan phức tạp không?
2. **Chi phí (Cost)**: Mỗi lần gọi LLM có giá bao nhiêu? Agent có thể gọi 10-20 lần cho một task
3. **Độ trễ (Latency)**: Agent mất bao lâu để "suy nghĩ"?

#### Các Lựa Chọn
- **Tối đa khả năng**: GPT-4, Claude - chất lượng cao nhất, chi phí cao, độ trễ lớn
- **Tối ưu tốc độ và chi phí**: Gemini Flash, Llama 3 - nhanh, rẻ, có thể mắc lỗi với plan phức tạp
- **Tối ưu riêng tư**: Self-hosted open source models - kiểm soát data hoàn toàn nhưng tốn chi phí infrastructure

## 2. The Senses - Cảm Quan (Perception Model)

### Vấn Đề
- LLM là text processing engine xuất sắc
- Thế giới thực không sạch sẽ: messy, unstructured, multimodal
- LLM không thể hiểu PDF 100 trang, JSON phức tạp, hình ảnh, audio trực tiếp

### Chức Năng: Normalization
- Dịch dữ liệu messy thành clean text mà LLM hiểu được
- Pipeline gồm các bộ dịch nhỏ hơn

### Quy Trình Xử Lý

| Input | Processor | Output |
|-------|-----------|--------|
| Hình ảnh (ảnh sản phẩm bị lỗi) | Vision Model | Text description: "T-shirt có lỗ thủng ở góc" |
| PDF 100 trang | Extraction Library | Structured text summary |
| Audio file | Speech-to-text model | Transcript text |
| JSON/API response | JSON parser | Clean text summary |
| User text (có typo) | Text processor | Clean normalized text |

### Ví Dụ Thực Tế
**Support Agent với 2 senses:**
1. User gõ: "my knee shirt has a hole in it" (có typo "knee" thay vì "new")
2. User upload ảnh chiếc áo bị lỗi
3. Perception model:
   - Text processor: sửa "knee" → "new"
   - Vision processor: phân tích ảnh, mô tả lỗ thủng
   - Gộp thành: "Customer reports their new shirt has a hole in it. [Image analysis: Visible tear at corner]"
4. LLM nhận được context hoàn chỉnh để suy luận

## 3. The Blueprint - Bản Thiết Kế (Planning Model)

### Vấn Đề Cần Giải Quyết
- User không đưa ra command đơn giản một bước
- User đưa ra **complex goal**: "Book my complete trip to London next week"
- Agent không thể đạt được goal này trong một action đơn lẻ

### Chức Năng: Task Decomposition
- Phân rã complex goal thành sequence của các simple, executable steps
- **Plan không phải là câu trả lời cuối cùng** - plan là data structure (list/graph)
- Plan là blueprint được chuyển cho Action Model

### Ví Dụ Plan
**Goal**: Book my trip to London

**Plan**:
1. Find available flights to London
2. Find hotels in London
3. Book selected flight
4. Book selected hotel
5. Send confirmation email

### Hai Phương Pháp Lập Kế Hoạch

#### Static Planning (Chain of Thought)
- Tạo toàn bộ plan trước khi thực hiện bất kỳ action nào
- Phù hợp: workflows có tính predictable, tuần tự (book flight → book hotel → send email)

#### Dynamic Planning (ReAct)
- Agent không lập plan toàn bộ từ đầu
- Plan chỉ **bước tiếp theo duy nhất**
- Execute bước đó → Quan sát kết quả → Quay lại reasoning để quyết định bước tiếp theo
- Vòng lặp: Reason → Act → Observe → Reason → Act...
- Đây là **màu sắc của modern agentic systems**

### System Prompt Cho Planning
```
You are a task decomposition engine.

Goal: [User's goal from perception]

Available Tools:
- get_weather(location)
- search_flights(origin, destination, date)
- book_flight(flight_id)
- send_email(to, subject, body)

Instruction: Create a step-by-step plan to achieve the goal using only the available tools.

Plan:
```

## 4. The Notebook - Bộ Nhớ (Memory Architecture)

### Vấn Đề: Groundhog Day Problem
- Không có memory → mỗi perception là sự kiện isolated, contextless
- Agent không nhớ những gì nói 3 giây trước
- Agent thực hiện step 1 của plan 5 bước → không nhớ để thực hiện step 2

### Hai Loại Memory

#### Short-term Memory (RAM / Working Memory)
- **Physically**: Context window của LLM
- **Chứa**: Current user query, plan vừa tạo, kết quả API call gần nhất, conversation history ngay lập tức
- **Volatile**: Bị giới hạn bởi context window size
- **Limitation**: Context Window Problem - không thể nhét toàn bộ history, 500-page report vào

#### Long-term Memory (Hard Drive / Persistent Memory)
- **Physically**: External vector database
- **Chứa**: All past conversations, user preferences (e.g., "user office is in New York", "user prefers window seat"), entire knowledge base
- **Permanent**: Dữ liệu được lưu trữ lâu dài
- **Slower**: Truy cập chậm hơn short-term

### Kết Nối: RAG (Retrieval Augmented Generation)
```
User Query → LLM realizes cần thông tin từ quá khứ
              ↓
         Search Long-term Memory (Vector DB)
              ↓
         Retrieve relevant chunks
              ↓
         Inject vào Short-term Memory (Context Window)
              ↓
         LLM generates response với context đầy đủ
```

### Loại Databases Cho Long-term Memory

| Type | Use Case | Examples |
|------|----------|----------|
| **Key-Value/Document DB** | User facts, preferences, chat history | Redis, MongoDB |
| **SQL Databases** | Enterprise relational data | PostgreSQL, MySQL |
| **Vector Databases** | Semantic search trong large knowledge bases | Pinecone, Weaviate, Chroma |

### Vector Database
- Lưu trữ high-dimensional vectors (embeddings)
- Cho phép **semantic search** - tìm kiếm theo ý nghĩa, không chỉ keyword
- Khi agent cần tìm thông tin từ 5,000 documents → dùng vector search

## 5. The Hands - Hành Động (Action & Tool Calling)

### Vấn Đề
- Plan chỉ là thought, là data structure
- Plan không phải là action thực sự
- Cần cơ chế để plan trở thành thay đổi thực tế trong thế giới thực

### Công Cụ (Tools)
- **Definition**: Any external function or API mà agent được phép sử dụng
- **Examples**: 
  - Local Python functions
  - API calls (weather, email, calendar)
  - Database queries
  - Code execution

### Function Calling Mechanism (3 Steps)

#### Step 1: Tool Definition
- Cung cấp JSON schema mô tả các công cụ sẵn có:
```json
{
  "tools": [
    {
      "name": "get_weather",
      "description": "Get weather for a location",
      "parameters": {
        "location": {"type": "string", "description": "City name"}
      }
    },
    {
      "name": "send_email", 
      "description": "Send an email",
      "parameters": {
        "to": {"type": "string"},
        "subject": {"type": "string"},
        "body": {"type": "string"}
      }
    }
  ]
}
```

#### Step 2: LLM Generates Tool Call
- Khi LLM cần dùng tool, output là JSON:
```json
{
  "tool_call": "get_weather",
  "arguments": {"location": "London"}
}
```
- LLM không thực thi tool - chỉ tạo request

#### Step 3: Action Model Executes
- Action Model intercept JSON request
- Parse và execute actual Python function
- Nhận kết quả từ tool
- Convert thành text và feedback vào perception layer

### Vòng Lặp ReAct Chi Tiết
```
1. LLM (Brain): Reason → "I need weather data"
                     ↓
              Generate: {"call": "get_weather", "args": {"location": "London"}}
                     ↓
2. Action Model: Parse JSON
                     ↓
              Execute: get_weather(location="London")
                     ↓
              Result: "25°C, cloudy"
                     ↓
3. Action Model: Convert to text → "Weather in London is 25°C and cloudy"
                     ↓
4. Perception: Feed vào LLM
                     ↓
5. LLM: Reason với dữ liệu mới → Generate final answer
```

## 6. Security & Guardrails

### Nguyên Tắc Cốt Lõi
> "With great power comes great responsibility"

### Thách Thức
- Agent là autonomous, non-deterministic system
- Khi cho agent quyền truy cập tools → như cho họ chìa khóa nhà
- **Nightmare scenario**: Agent được cho tool "delete_user_records" để xóa test data → nhầm lẫn xóa production database

### Architectural Guardrails

#### 1. Least Privilege
- Không bao giờ cho agent master key
- Weather agent chỉ có quyền "get_weather"
- Không có quyền "send_email" hay "delete_database"

#### 2. Read-only vs Write Access
- Read operations (get_weather, search_data): Generally safe
- Write operations (delete_record, send_email, transfer_money): Inherently dangerous

#### 3. Human-in-the-Loop (HITL)
- Cho các sensitive/destructive actions:
  - Sending email
  - Deleting data
  - Spending money
- Agent cannot act autonomously
- Plan được trình bày cho human operator để approve/deny
- Đây là "ultimate guardrail"
