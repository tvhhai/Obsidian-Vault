# Các Design Patterns Trong AI Agentic

## I. Single Agent Patterns

### 1. Tool Use Pattern (Sử Dụng Công Cụ)

#### Mô Tả
- Agent có quyền truy cập và sử dụng các công cụ bên ngoài (external functions/APIs)
- Tool đóng vai trò "bàn tay" của agent - thực hiện thay đổi thực tế trong thế giới thực

#### Ví Dụ
```
User: "Thời tiết ở Paris thế nào?"

Agent (không có tool):
- "Paris thường có thời tiết ôn hòa vào mùa xuân..." (thông tin chung chung)

Agent (có get_weather tool):
- Gọi get_weather(location="Paris")
- Nhận: "15°C, cloudy, 60% humidity"
- Trả lời: "Hiện tại ở Paris là 15°C, trời nhiều mây"
```

#### Thành Phần
- **Tool Schema**: Mô tả công cụ và parameters
- **Function Calling**: LLM tạo JSON request
- **Execution**: Action model thực thi công cụ

### 2. Planning Pattern (Lập Kế Hoạch)

#### Mô Tả
- Agent phân rã (decompose) mục tiêu phức tạp thành chuỗi các bước đơn giản
- Plan là data structure (list/graph) chứ không phải câu trả lời cuối

#### Hai Phương Pháp

##### Static Planning
- Tạo toàn bộ plan trước khi thực hiện
- Phù hợp workflows predictable, tuần tự

##### Dynamic Planning (ReAct-style)
- Plan chỉ bước tiếp theo
- Execute → Observe → Re-plan

#### Ví Dụ
```
Goal: "Book my trip to London"

Plan:
Step 1: Find flights (search_flights)
Step 2: Find hotels (search_hotels)
Step 3: Book selected flight (book_flight)
Step 4: Book selected hotel (book_hotel)
Step 5: Send confirmation (send_email)
```

### 3. Reflection Pattern (Metacognition) - Tự Phê Bình

#### Mô Tả
- Agent có một "critic" component đánh giá kết quả của chính mình
- Tự kiểm tra, tự cải thiện output

#### Quy Trình
```
1. Generate: Agent tạo output ban đầu
2. Reflect: Critic đánh giá output
   - "Is this correct?"
   - "What's wrong with this?"
   - "How could this be improved?"
3. Improve: Agent sửa chữa dựa trên feedback
4. (Optional) Repeat cho đến khi đạt chất lượng
```

#### Ví Dụ: Code Generation
```
Task: Write a function to calculate fibonacci

Generate:
def fib(n):
    if n <= 1:
        return n
    return fib(n-1) + fib(n-2)

Reflect (Critic):
"This function is correct but has exponential time complexity O(2^n).
For large n, this will be very slow. Should use memoization or iterative approach."

Improve:
def fib(n, memo={}):
    if n in memo:
        return memo[n]
    if n <= 1:
        return n
    memo[n] = fib(n-1, memo) + fib(n-2, memo)
    return memo[n]
```

#### Lợi Ích
- Tự động cải thiện chất lượng output
- Giảm lỗi không cần thiết
- Metacognition - "thinking about thinking"

### 4. ReAct Compound Pattern

#### Mô Tả
- **ReAct** = **Re**ason + **Act**
- Là compound pattern kết hợp Planning + Tool Use + Observation

#### Quy Trình Vòng Lặp
```
Thought (Suy luận) → Action (Hành động) → Observation (Quan sát) → Repeat
```

#### Chi Tiết Từng Bước

##### Step 1: Thought (Reason)
- LLM suy luận về tình huống hiện tại
- Quyết định cần làm gì tiếp theo
- Chỉ plan cho **bước tiếp theo duy nhất**

##### Step 2: Action
- Sử dụng tool để thực hiện bước đó
- Generate JSON function call

##### Step 3: Observation
- Nhận kết quả từ tool
- Feed kết quả vào context

##### Step 4: Repeat
- Suy luận lại với thông tin mới
- Quyết định bước tiếp theo

#### Ví Dụ Thực Tế
```
User: "What's the weather like in London and should I take an umbrella?"

Loop 1:
Thought: "I need weather information to answer. I should call get_weather."
Action: {"tool": "get_weather", "args": {"location": "London"}}
Observation: "Temperature: 15°C, Conditions: Light rain"

Loop 2:
Thought: "London has light rain. User should take umbrella. I have enough info."
Action: {"response": "It's currently 15°C with light rain in London. Yes, you should take an umbrella!"}
Observation: (Task complete)
```

#### Lợi Ích
- **Adaptive**: Thích ứng với kết quả thực tế từ mỗi bước
- **Robust**: Xử lý được failure ở từng bước
- **Dynamic**: Không cần plan tất cả từ đầu

#### Trade-off
- **Chậm hơn**: Mỗi step đều cần LLM call
- **Đắt hơn**: Nhiều round trips đến LLM
- Nhưng: **Đáng tin cậy hơn** vì có thể phản ứng với environment

## II. Multi-Agent Patterns

### 1. Router & Specialist Pattern (Bộ Định Tuyến và Chuyên Gia)

#### Mô Tả
- **Router Agent**: Nhận request và quyết định chuyển đến specialist nào
- **Specialist Agents**: Mỗi agent chuyên về một domain cụ thể

#### Kiến Trúc
```
                    User Request
                          ↓
                    Router Agent
                   (Phân tích intent)
                          ↓
        ┌─────────────────┼─────────────────┐
        ↓                 ↓                 ↓
   Tech Support      Billing Agent      Sales Agent
   (Hỗ trợ kỹ thuật)  (Thanh toán)      (Bán hàng)
        ↓                 ↓                 ↓
   Giải quyết vấn đề  Xử lý hoàn tiền   Tư vấn sản phẩm
```

#### Ví Dụ: Customer Support System
```
User: "Tôi muốn hoàn tiền cho đơn hàng #12345"

Router Agent phân tích:
- Intent: refund_request
- Entities: order_id=12345
- Routing: → Billing Agent

Billing Agent xử lý:
- Verify order #12345
- Check refund policy
- Process refund
- Confirm với user
```

#### Lợi Ích
- **Modularity**: Mỗi specialist đơn giản, dễ bảo trì
- **Scalability**: Thêm specialist mới không ảnh hưởng hiện tại
- **Expertise**: Mỗi agent tối ưu cho domain riêng

---

### 2. Handoff Pattern (Sequential Workflow)

#### Mô Tả
- Task được chuyển tiếp tuần tự giữa các agents
- Mỗi agent hoàn thành phần việc rồi chuyển cho agent tiếp theo

#### Kiến Trúc
```
User Request
    ↓
Agent 1 (Intake/Triage)
    ↓ (handoff với context)
Agent 2 (Processing)
    ↓ (handoff với kết quả)
Agent 3 (Verification)
    ↓ (handoff với xác nhận)
Agent 4 (Completion)
    ↓
Final Response to User
```

#### Ví Dụ: Content Creation Pipeline
```
User: "Viết một bài blog về AI Agentic"

Agent 1 - Researcher:
- Thu thập thông tin về AI Agentic
- Tạo outline
- Handoff → Writer

Agent 2 - Writer:
- Viết draft dựa trên research
- Tạo content đầy đủ
- Handoff → Editor

Agent 3 - Editor:
- Review và sửa lỗi
- Cải thiện flow và clarity
- Handoff → Publisher

Agent 4 - Publisher:
- Format cho blog platform
- Thêm metadata
- Publish
```

#### Key Consideration: State Transfer
- Thông tin và context phải được chuyển đầy đủ giữa các agents
- Shared state hoặc message passing mechanism

### 3. Group Chat / Debate Pattern

#### Mô Tả
- Nhiều agents thảo luận, tranh luận để đạt được consensus hoặc giải pháp tối ưu
- Mô phỏng "round table discussion"

#### Kiến Trúc
```
    ┌─────────┐
    │  Moderator  │ (Điều phối cuộc thảo luận)
    └────┬────┘
         │
    ┌────┼────┐
    ↓    ↓    ↓
Agent A Agent B Agent C
(Cấp tính) (Trung lập) (Bảo thủ)
    └────┬────┘
         ↓
    Consensus / Best Solution
```

#### Ví Dụ: Investment Decision
```
Task: Quyết định có nên đầu tư vào startup X?

Agent 1 - Optimist (Lạc quan):
"Thị trường đang tăng trưởng 30%/năm. Team có kinh nghiệm. Nên đầu tư."

Agent 2 - Pessimist (Bi quan):
"Đối thủ cạnh tranh lớn. Burn rate cao. Rủi ro quá lớn. Không nên đầu tư."

Agent 3 - Analyst (Phân tích):
"Dựa trên số liệu: P/E ratio tốt. Nhưng cần due diligence thêm về IP."

Moderator tổng hợp:
"Recommendation: Conditional approval. Đầu tư với điều kiện IP audit passed."
```

#### Lợi Ích
- **Diverse perspectives**: Nhiều góc nhìn khác nhau
- **Balanced decisions**: Giảm bias của single agent
- **Creative solutions**: Brainstorming giữa các agents

### 4. Swarm Pattern (Parallelization)

#### Mô Tả
- Nhiều agents thực hiện **song song** cùng một task hoặc sub-tasks
- Kết quả được tổng hợp (aggregate)

#### Kiến Trúc
```
         Task
           ↓
    ┌──────┴──────┐
    ↓      ↓      ↓
 Agent1 Agent2 Agent3
    ↓      ↓      ↓
 Result1 Result2 Result3
    └──────┬──────┘
           ↓
    Aggregator Agent
           ↓
    Final Result
```

#### Ví Dụ 1: Code Review
```
Task: Review một PR

Agent 1 - Security Reviewer:
- Kiểm tra vulnerabilities

Agent 2 - Performance Reviewer:
- Kiểm tra efficiency

Agent 3 - Style Reviewer:
- Kiểm tra code style

(Tất cả chạy song song)

Aggregator:
- Gộp tất cả comments
- Phân loại theo mức độ nghiêm trọng
- Tạo summary report
```

#### Ví Dụ 2: Research
```
Task: Tìm hiểu về "renewable energy trends 2024"

Agent 1: Tìm kiếm từ academic sources
Agent 2: Tìm kiếm từ news sources  
Agent 3: Tìm kiếm từ industry reports
Agent 4: Tìm kiếm từ social media trends

Aggregator: Tổng hợp thành comprehensive report
```

#### Lợi Ích
- **Speed**: Nhiều tasks chạy song song nhanh hơn tuần tự
- **Comprehensive**: Kết quả từ nhiều nguồn/góc độ
- **Resilience**: Một agent fail không ảnh hưởng toàn bộ

## III. Control & Safety Patterns

### 1. Human-in-the-Loop (HITL)

#### Mô Tả
- Con người tham gia vào vòng lặp quyết định
- Agent không thể tự động thực hiện các hành động nhạy cảm

#### Áp Dụng Cho
- **Destructive actions**: Xóa dữ liệu, hủy tài khoản
- **Financial transactions**: Chuyển tiền, thanh toán
- **External communications**: Gửi email, đăng bài
- **High-risk decisions**: Y tế, pháp lý

#### Quy Trình
```
Agent Plan
    ↓
Human Review
    ↓ (Approve / Modify / Reject)
Execution
    ↓
Feedback to Agent
```

#### Implementation
```
if action.sensitivity_level == "high":
    send_for_approval(agent_plan)
    await human_response()
    if human_response == "approve":
        execute(action)
    else:
        revise_plan()
```

---

### 2. Ejection Pattern (Custom Logic)

#### Mô Tả
- Cho phép interject logic tùy chỉnh vào agent flow
- Escalate hoặc xử lý edge cases đặc biệt

#### Ví Dụ
```
Agent đang xử lý customer request
    ↓
Detect: "Customer mention 'lawsuit' or 'lawyer'"
    ↓
EJECT: Bypass normal flow
    ↓
Escalate to Legal Team immediately
```

## IV. Nguyên Tắc Thiết Kế Agentic

### 4 Nguyên Tắc Cốt Lõi

| Principle | Mô Tả | Vai Trò |
|-----------|-------|---------|
| **Goal-directed** | Định nghĩa rõ mục tiêu agent cần đạt | Định hướng |
| **Reactive** | Phản ứng với thay đổi trong environment | Tương tác |
| **Stateful** | Duy trì bộ nhớ qua thời gian | Liên tục |
| **Autonomous** | Tự đưa ra quyết định và hành động | Độc lập |

### Mối Quan Hệ
```
Goal (Đích đến) + Reactive (Nhìn đường) + Stateful (Nhớ đã đi đâu) = Autonomy (Tự lái)
```
